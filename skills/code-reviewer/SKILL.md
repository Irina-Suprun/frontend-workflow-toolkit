# Code Reviewer

Проводить структуроване рев'ю diff'а або набору файлів і видає звіт трьома
рівнями серйозності. Це скіл ДЛЯ Claude — інструкції нижче виконуються, а
не переказуються користувачу.

## Коли викликається

1. **Явно** — людина просить ревью коду напряму.
2. **Напівавтоматично з `git-workflow`** — перед тим, як `git-workflow`
   згенерує опис Pull Request (Крок 5 у `git-workflow`), він застосовує
   критерії цього скіла до diff'а. Це не hard-блокування на рівні
   інструменту (немає окремого технічного PreToolUse hook для цього
   кроку) — це логічний крок усередині самого `git-workflow`, який
   виконується щоразу перед генерацією PR-опису.

## Класифікація знахідок

### 🔴 Critical — блокує PR, доки не виправлено (або людина явно не погодилась ігнорувати)

Сюди потрапляє лише те, що реально ламає стандарти проєкту чи є ризиком:

- `any` в TypeScript (включно з неявним `any` через відсутню типізацію параметра)
- `enum` замість union type / `as const` об'єкта
- Magic values замість токенів теми в SCSS — хардкод hex-кольору (`#ff0000`),
  довільний px для spacing/radius/шрифту замість змінної з `_theme.scss`
- Повна відсутність семантики там, де вона критична — `<div onClick>` замість
  `<button>`, клікабельний елемент без будь-якого способу дістатись до нього
  з клавіатури
- Натяки на API-ключі, токени, паролі чи секрети в коді — негайно
  позначай як критичне і замасковуй значення в самому звіті
- Redux-стан або бізнес-логіка, написана прямо всередині UI-компонента
  (порушення розділення відповідальності)
- Jest-тести відсутні для нової бізнес-логіки (нового екшена, селектора,
  хука з логікою) — сама відсутність тесту тут критична, а не рекомендація

### 🟡 Suggestions — не блокує, але варто зробити

- Відсутня мемоізація (`useMemo`/`useCallback`) там, де обчислення чи
  функція реально дорогі або передаються в дочірній компонент, що
  ре-рендериться
- Точкові a11y-покращення понад мінімум — `aria-label` для іконки-кнопки
  без видимого тексту, коли клавіатурна навігація і фокус вже коректні
- Дрібні покращення читабельності назв змінних/функцій
- Можливість винести повторювану логіку в спільний хук

### 🟢 Passed Checks — коротко підтверди, що саме перевірено і пройдено

Обери з цього списку те, що релевантне конкретному diff'у (не вигадуй
пункти, які не стосуються цих файлів):

- Colocation файлів (`Component.tsx` + `Component.types.ts` +
  `Component.module.scss` + `index.ts` поруч)
- `No any found` / `No enum found`
- Типізовані Redux-хуки (`useAppDispatch`/`useAppSelector`), а не
  сирі `useDispatch`/`useSelector`
- Jest-тести додані і покривають новий кейс
- `:focus-visible` та коректний `tabindex` там, де є клавіатурна навігація
- LF-кінці рядків (якщо це видно з diff'а)

## Формат звіту

Завжди видавай звіт саме в цій структурі — заголовки, емодзі та порядок
секцій фіксовані:

```markdown
### 🔍 Code Review Report

#### 🔴 Critical Issues (Нужно исправить до PR):
- `File.tsx`: [що знайдено, рядок якщо відомий] → [конкретна заміна/фікс]

#### 🟡 Suggestions (Рекомендации):
- `File.ts`: [що можна покращити] → [конкретна пропозиція]

#### 🟢 Passed Checks:
- [x] [Пройдена перевірка]
```

Якщо в якійсь секції немає знахідок — залиш заголовок і напиши одним
рядком "Немає" замість порожньої секції; не приховуй секцію повністю.

Пиши файлові шляхи та назви змінних як у коді (без перекладу), решту
тексту звіту — тією ж мовою, якою людина веде розмову.

## Правило зупинки при критичних знахідках

Якщо є хоча б одна 🔴 Critical позиція — **не продовжуй одразу до опису
PR**. Спочатку виведи звіт повністю, а потім задай пряме питання:

> "В коде есть N критических замечаний. Исправим их перед созданием PR?"

де N — точна кількість 🔴-пунктів. Дочекайся відповіді:

- Якщо людина погоджується виправити — допоможи виправити конкретні файли,
  потім повторно прогони перевірку на оновлений код перед тим, як
  `git-workflow` продовжить до опису PR.
- Якщо людина явно каже продовжити без виправлень ("ні, генеруй PR так",
  "залиш як є") — поважай це рішення і передай керування назад
  `git-workflow` для генерації опису PR; не наполягай повторно в цій же
  розмові.

Якщо критичних знахідок немає — просто покажи звіт (секції 🟡 і 🟢) і
без додаткових питань дозволь `git-workflow` продовжити до опису PR.

## Межі скіла

- Скіл лише аналізує та звітує — сам ніколи не переписує файли без
  прямого прохання людини виправити конкретну знахідку.
- Скіл ніколи не виконує `git`- чи `npm`-команди — рев'ю робиться по
  вже наявному в контексті diff'у/коду, а не шляхом запуску лінтера.
  Якщо потрібен саме запуск ESLint/Jest — це вже сфера Auto Mode hook'а
  (`hooks/scripts/auto-lint-test.py`), а не цього скіла.
- Не позначай як критичне те, що є справою стилю чи смаку без порушення
  конкретного стандарту проєкту — критичний рівень зарезервований для
  реальних порушень правил, перелічених вище.
