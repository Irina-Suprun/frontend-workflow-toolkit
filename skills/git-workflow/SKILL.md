# Git Workflow: генератор commit-повідомлень та описів PR

## Призначення

Перетворює сирі зміни в коді (diff, `git status`, або звичайний опис того, що змінилося) на два готові до копіювання артефакти:

1. **Commit-повідомлення** у форматі Conventional Commits з gitmoji-префіксом — зазвичай одне; кілька пропонується лише коли diff явно змішує непов'язані речі (див. Крок 6).
2. **Структурований опис Pull Request**, який дає рев'юєру все необхідне без потреби перечитувати diff рядок за рядком.

Людина, яка користується цим скілом, працює у **ручному режимі**: ніколи не виконуй `git`-команди, не пропонуй `git commit`/`git push`/`git add` і не пропонуй сама створити або відкрити PR на будь-якій платформі. Твоя робота закінчується на тому, що людина скопіює текст сама. Це стосується навіть якщо вхідні дані виглядають як з живої терміналної сесії — сприймай це як вставлений текст, а не як репозиторій, з яким можна щось робити.

## Крок 1: Зрозуміти, що насправді змінилося

Перш ніж щось писати, прочитай diff (або опис) достатньо уважно, щоб відповісти на чотири питання:

- **Який основний намір?** Не просто "ці рядки змінились", а *чому* — нова можливість, фікс, візуальне уточнення, страховка (тести), внутрішнє прибирання коду.
- **Це одна логічна зміна чи кілька, зшитих разом?** Зафіксуй це для себе, але не дій одразу — скільки насправді потрібно окремих комітів вирішується в Кроці 7, не тут.
- **Чи щось тут змінює публічну поведінку так, що це є breaking change?** Змінена сигнатура функції, видалений пропс, перейменований Redux-екшн або змінена форма відповіді API — усе це вважається breaking навіть якщо людина не позначила це сама. Обов'язково повідом про це у відповіді.
- **Чи є за цією зміною задача в Linear?** Бренчі в цьому робочому процесі створюються з номером задачі, вбудованим у назву (напр. `eng-482-fix-stale-token`), тож перевір, чи повідомлення людини, назва бренча або сам diff уже містить ключ формату `[A-Z]{2,5}-\d+`. Якщо ключ не знайдено — **запитай, перш ніж писати щось інше**: одне пряме питання типу "Який номер задачі в Linear для цього — чи цієї зміни немає в трекері?". Продовжуй без ключа лише якщо людина підтвердила, що задачі справді немає.

Якщо вхідних даних замало для відповіді на ці питання (наприклад, лише список файлів без змісту) — задай одне цільове уточнююче питання, а не вигадуй намір. Об'єднай його з питанням про Linear вище, щоб людину питали лише раз.

## Крок 2: Визначити тип зміни (Conventional Commits)

| Тип | Gitmoji | Коли використовувати |
|---|---|---|
| `feat` | ✨ | Нова можливість або поведінка, помітна користувачу |
| `fix` | 🐛 | Виправлення багу — щось працювало некоректно, а тепер працює правильно |
| `refactor` | ♻️ | Внутрішня перебудова без зміни поведінки (перейменування, виділення хуків, перенесення логіки з компонентів у Redux тощо) |
| `style` | 💄 | Форматування, пробіли, візуальні SCSS-правки без зміни логіки |
| `test` | ✅ | Додавання чи оновлення лише Jest-тестів |
| `docs` | 📝 | README, коментарі-документація, JSDoc |
| `chore` | 🔧 | Інструментарій, конфіги, оновлення залежностей, CI — нічого, що потрапляє до користувача |
| `perf` | ⚡️ | Зміна, зроблена спеціально для покращення продуктивності |

Обери рівно один тип на коміт. Якщо зміна дійсно охоплює два типи одразу (наприклад, фікс, який також додає регресійний тест), перемагає домінантний намір, орієнтований на користувача — фікс з тестом — це все ще `fix`, не `test`.

Якщо зміна є breaking (див. Крок 1), додай `💥` одразу після емодзі типу і познач тип знаком `!` — наприклад, `✨💥 feat!(cart): ...` — і завжди додавай футер `BREAKING CHANGE:` з поясненням, що саме зламається і як мігрувати.

## Крок 3: Визначити scope

Scope у Conventional Commits вказується у дужках: `feat(auth): ...`. Визначай його зі шляхів файлів, використовуючи ці евристики (налаштовані під кодову базу на React / Next.js / Redux Toolkit / TypeScript / SCSS-модулях):

- `src/features/<назва>/**` або `src/modules/<назва>/**` → scope — `<назва>`
- `store/slices/<назва>Slice.ts`, `redux/<назва>/**`, або будь-який файл слайсу/селектора Redux Toolkit → scope — доменна назва слайсу (напр. `cartSlice.ts` → `cart`)
- `components/<НазваКомпонента>/**` (за структурою `Component.tsx` + `Component.types.ts` + `Component.module.scss` + `index.ts`) → scope — kebab-case назва компонента (напр. `BookingForm` → `booking-form`)
- `hooks/use<Назва>.ts` → scope — `hooks`, якщо хук не прив'язаний вузько до однієї фічі — тоді використай назву цієї фічі
- Змінилися лише `*.module.scss` чи інші стильові файли, без логіки → scope — `styles`
- Змінилися лише `*.test.ts(x)` → тип залишається `test`, scope — те, що тестується (напр. `cartSlice.test.ts` → `cart`)
- Зміни охоплюють кілька непов'язаних верхньорівневих ділянок → це зазвичай сигнал, що коміт варто розбити (див. Крок 1). Якщо людина підтверджує, що хоче один коміт — просто пропусти scope, а не вигадуй його.

Коли є сумніви, обирай найвужчий scope, який усе ще точний — `(cart)` кращий за `(store)`, якщо змінився лише cart-слайс.

## Крок 4: Написати commit-повідомлення

Формат:

```
<gitmoji> <type>(<scope>): <imperative, present-tense summary, lowercase, no trailing period>

<optional body: what changed and why, wrapped ~72 chars, blank line above>

<optional footer: BREAKING CHANGE: description of the break>
<optional footer: Refs ENG-123 / Closes ENG-123>
```

Commit-повідомлення завжди пишеться англійською — це стандарт для Git-історії незалежно від мови спілкування з людиною.

Правила для рядка теми (subject):
- Наказова форма (imperative): "add", "fix", "extract" — не "added", "adds", "fixed"
- ≤ 72 символи, без крапки в кінці (gitmoji не рахується в цей ліміт — це візуальний префікс, а не частина речення)
- Описує *ефект* зміни, а не механіку ("fix focus loss on Safari input blur", а не "change useEffect dependency array")

Додавай тіло (body) лише якщо *чому* не очевидне з самого заголовка — не роздувай кожен коміт переказом diff.

**Посилання на Linear у футері:** на цьому кроці в тебе вже має бути ключ задачі (або явне підтвердження, що задачі немає) з Кроку 1. Використовуй `Closes ENG-482` лише коли цей коміт повністю закриває задачу; використовуй `Refs ENG-482`, коли зміна пов'язана, але не завершує роботу (безпечніший варіант за замовчуванням, якщо не впевнена). Якщо людина підтвердила, що задачі немає — просто не додавай футер.

**Приклад:**
Вхідні дані: змінено редʼюсер `authSlice` так, що невдалий логін очищує збережений токен, плюс додано Jest-тест на цей кейс. Бренч: `eng-482-fix-stale-token`.
Результат:
```
🐛 fix(auth): clear stored token on failed login

Previously a failed login attempt left a stale token in state,
which could let the UI briefly render as authenticated.

Closes ENG-482
```

## Крок 5: Авто-ревʼю перед описом PR (`code-reviewer`)

Перш ніж переходити до Кроку 6 (опис PR), застосуй скіл `code-reviewer`
до цього ж diff'а. Це стосується лише опису PR — commit-повідомлення з
Кроку 4 вже написане і не залежить від результату ревʼю.

1. Виконай ревʼю за критеріями `code-reviewer` і виведи звіт у форматі
   `### 🔍 Code Review Report` (секції 🔴 / 🟡 / 🟢).
2. Якщо є хоча б одна 🔴 Critical позиція — зупинись і запитай:
   "В коде есть N критических замечаний. Исправим их перед созданием PR?"
   Не генеруй опис PR у цьому ж повідомленні — дочекайся відповіді.
   - Людина погодилась виправити → допоможи виправити, потім повтори
     ревʼю на оновленому коді, і лише після чистого результату переходь
     до Кроку 6.
   - Людина явно попросила продовжити без виправлень → переходь до
     Кроку 6 без додаткових питань.
3. Якщо критичних позицій немає — покажи звіт (🟡 і 🟢) і одразу переходь
   до Кроку 6 в тому самому повідомленні, без зупинки на підтвердження.

## Крок 6: Написати опис PR

Завжди використовуй саме цю структуру. Вона потрібна, щоб рев'юєр (або сама людина в майбутньому) міг оцінити зміну без повторного відкриття кожного файлу:

```markdown
## 🔗 Linear issue
[ENG-123 — or "N/A" if genuinely nothing to link]

## 📝 Summary
[1-3 sentences: what changed and why, in plain language — the motivation, not a file list]

## 🏷️ Type of change
- [ ] ✨ New feature
- [ ] 🐛 Bug fix
- [ ] 💥 Breaking change
- [ ] ♻️ Refactoring

## 🔍 Key Changes
- [Bullet per meaningful change, grouped by area if there are several — not a line-by-line diff narration]

## 🧪 How to test
1. [Concrete, numbered steps a reviewer or QA person can actually follow: where to click, what input to give, what result to expect]

## 📸 Screenshots
[Include only when the change is visual (UI, layout, styling). Leave a placeholder line like `[attach before/after screenshot]` for the person to fill in manually — never invent or fake a screenshot. Omit this whole section for purely logic/data changes (e.g. a Redux slice with no UI surface).]

## ♿️ Accessibility / Performance notes
[State plainly whether this PR touches a11y or performance, and how. If it doesn't, say "No a11y or performance impact." — don't skip the section, an explicit "none" is more useful than silence.]
```

Сам опис PR (заголовки секцій, текст усередині) пишеться англійською — так само, як і commit-повідомлення, це стандарт для команди рев'юєрів.

Пояснення, як добре заповнити кожну секцію:

- **Linear issue**: на цьому етапі ти вже запитала про це в Кроці 1, тож просто впиши те, що з'ясувала — ключ задачі (напр. `ENG-482`) або, якщо людина підтвердила відсутність задачі, `N/A`. Ніколи не пиши тут плейсхолдер — ця секція має бути вирішена ще до того, як ти дійшла сюди.
- **Summary** відповідає на питання "чому цей PR існує", а не "які файли змінилися". Рев'юєр повинен зрозуміти мотивацію ще до читання коду.
- **Type of change**: чесно відзначай галочки — PR, що одночасно фіксить баг і рефакторить навколишній код, має обидві позначки.
- **Key Changes** — це список ключових пунктів, а не переказ diff. Групуй пов'язані зміни файлів в один пункт ("Extracted date-formatting logic into a `useFormattedDate` hook and removed the duplicated logic from three components"), а не перелічуй кожен файл.
- **How to test** мають бути виконувані людиною, яка ніколи не бачила код — називай конкретні сторінки, вхідні дані, очікувані результати. Для зміни в Redux-слайсі чи селекторі без UI-поверхні цілком підходить "run `npm test` and confirm the new cases in `cartSlice.test.ts` pass".
- **Screenshots**: ніколи не вигадуй і не описуй фейковий скріншот. Твоя задача — лише розпізнати, коли він очікується, і залишити плейсхолдер — реальне зображення додає сама людина.
- **Accessibility / Performance notes** — активно шукай тригери: будь-яка зміна, що торкається керування фокусом, клавіатурної навігації, ARIA-атрибутів, семантичного HTML, контрасту/візуального стану, розміру бандла, мемоізації (`useMemo`/`useCallback`), частоти ре-рендерів чи рендерингу великих списків. Якщо щось із цього торкнулося зміни — назви це конкретно ("Added `:focus-visible` styles and corrected tab order in the modal" краще за "improved accessibility").

## Крок 7: Обробка кількох непов'язаних змін в одному diff

Diff часто несе більше одного наміру, але розбиття на кілька повних commit-блоків за замовчуванням радше заважає, ніж допомагає — сприймай це як те, на що варто звернути увагу, а не те, що варто нав'язувати.

**Поведінка за замовчуванням:** пиши одне commit-повідомлення, яке чесно відображає домінантну зміну, і один опис PR, що охоплює все (секція "Key Changes" — саме те місце, де кілька ниток показуються як окремі пункти). Якщо зміни достатньо пов'язані, щоб цілком підходити під один коміт (наприклад, фікс і його регресійний тест, або фіча і конфіг-правка, яка їй потрібна) — просто зроби так, без жодних коментарів.

**Коли справді запропонувати розбиття:** лише коли diff явно містить непов'язані речі, які не належать до однієї історії — типово 3 або більше окремих ділянок, або будь-яке поєднання реальної зміни коду з непов'язаним інструментальним/конфігураційним шумом. У такому випадку все одно веди з одним об'єднаним комітом + PR як основним результатом, а потім додай один короткий рядок наприкінці з пропозицією альтернативи: "Якщо хочеш, можу розбити це на окремі коміти (наприклад: фікс фокуса / cart-слайс / eslint) — просто скажи." Ніколи не видавай три окремі повні commit-блоки без запиту — спочатку запропонуй, розбивай лише якщо попросили.

## Тон та формат виводу

Пиши як senior-розробниця, що документує власну роботу для команди, яка читатиме це пізніше: точно, без пафосу, без слів-заповнювачів ("comprehensive", "robust") якщо вони не буквально правдиві. Виводь commit-повідомлення та опис PR як окремі готові до копіювання блоки коду — не обгортай їх зайвими коментарями, які людині доведеться видаляти перед тим, як вставити текст у Git чи інструмент для PR.

Якщо вхідних даних не вистачає для якогось поля шаблону (наприклад, немає способу визначити кроки тестування) — пиши `[needs input: ...]` на цьому місці, а не вигадуй правдоподібну, але хибну деталь.
