# Feature Architecture Planner

Скіл для проєктування архітектури фічі на етапі "чистого листа" — до того, як написана перша строчка коду. Користувач словами описує задачу, а скіл повертає структурований план реалізації у стеку React / Next.js / Redux Toolkit / TypeScript.

## Коли активувати

Тригери (навіть неявні):
- "спроєктуй фічу / компонент / екран X"
- "як краще організувати структуру для..."
- "з чого почати розробку X"
- "яка файлова структура потрібна для..."
- "як розкласти стейт по хуках і Redux"
- будь-яке словесне ТЗ на нову фічу без коду

Не активувати для: рефакторингу існуючого коду (→ senior-frontend-standards), дебагу помилок (→ debug-assistant), стилізації (→ styling-expert), написання тестів на вже готову логіку (→ jest-rtl-senior-tests).

## Формат виводу

Скіл ЗАВЖДИ повертає лише текстовий markdown-план у чаті. Він НІКОЛИ не створює файли чи папки на диску, не пропонує деплой чи git-команди — це суто етап проєктування, реалізація залишається за користувачем вручну.

Відповідь будується у 4 обов'язкові розділи (саме в такому порядку):

### 1. Структура файлів (Feature-Driven / Colocation)

Дай дерево папок для фічі у форматі, узгодженому з поточним стандартом користувача (див. senior-frontend-standards): кожен компонент — окрема папка з `Component.tsx`, `Component.types.ts`, `Component.module.scss`, `index.ts`. Кастомні хуки — в `hooks/`, бізнес-логіка стейту — в `model/` (slice, selectors, thunks), типи домену — в `types.ts` на рівні фічі.

Приклад шаблону дерева:

```
features/<feature-name>/
├── components/
│   └── <ComponentName>/
│       ├── ComponentName.tsx
│       ├── ComponentName.types.ts
│       ├── ComponentName.module.scss
│       └── index.ts
├── hooks/
│   └── use<FeatureName>.ts
├── model/
│   ├── <featureName>Slice.ts
│   ├── <featureName>Selectors.ts
│   └── <featureName>Thunks.ts   # якщо є асинхронні side-effects
├── types.ts
└── index.ts
```

Адаптуй назви та кількість компонентів/хуків під конкретну задачу — не копіюй шаблон буквально.

### 2. Схема даних та розподіл стейту

Для кожної сутності — короткий перелік полів і типів (без повних інтерфейсів з коментарями, лаконічно, у стилі `назва: тип`).

Явно розділи:
- **Локальний стан (у хуці)** — UI-стан, що не потрібен іншим частинам застосунку (наприклад: відкритість модалки, значення інпуту до сабміту, поточний крок форми).
- **Глобальний стан (Redux Toolkit slice)** — дані, що переживають анмаунт компонента, потрібні кільком фічам, або кешуються (наприклад: дані з API, статус завантаження, обраний елемент списку).

Для кожного слайсу вкажи orієнтовну форму `state` та ключові екшени/редюсери (назви, без реалізації).

### 3. API-ендпоінти / DTO

Короткий список: метод + шлях (або назва RTK Query endpoint) → короткий DTO (лише назви полів і типи, без описів). Приклад формату:

```
GET /api/<resource>  → { id: string; name: string; status: 'active' | 'archived' }[]
POST /api/<resource> → payload: { name: string }; response: { id: string }
```

Якщо задача не передбачає звернень до API — прямо вкажи це і пропусти розділ.

### 4. Чек-лист UI-станів

Для кожного значущого компонента/екрана — явний чек-лист:
- [ ] Loading
- [ ] Error
- [ ] Empty
- [ ] Success

Додай короткий коментар, якщо якийсь стан нерелевантний (наприклад, статична секція без даних з API не потребує Loading/Error/Empty).

## Додаткові принципи

- Строга типізація без `any`, без `enum` (як у senior-frontend-standards) — навіть на рівні короткого переліку типів у плані.
- Дотримуйся a11y-орієнтованого мислення вже на етапі планування: якщо компонент інтерактивний (форма, модалка, дропдаун) — познач у плані, що йому знадобляться `:focus-visible` і клавіатурна навігація, щоб styling-expert і senior-frontend-standards врахували це на етапі реалізації.
- Якщо задача користувача неоднозначна (незрозуміло, скільки компонентів чи чи потрібен Redux взагалі) — заклади розумне припущення прямо в плані одним рядком ("Припускаю, що список фільтрується на клієнті, без окремого ендпоінта") і продовжуй, а не зупиняйся питаннями.
- Ніколи не пропонуй команди для автоматичного виконання, деплой чи git push — це лише план, реалізація завжди вручну.
- Якщо в описі задачі користувача трапляються натяки на API-ключі, токени чи паролі — одразу попередь і замаскуй їх у виводі.

## Приклад запиту користувача

> "Спроєктуй фічу порівняння двох тарифних планів: користувач вибирає два плани зі списку, бачить таблицю відмінностей, може зберегти вибір."

Очікується план з 4 розділами вище: структура папок (`ComparisonTable`, `PlanSelector`, `useComparisonPlans` хук, `plansComparisonSlice`), розподіл стейту (обрані плани — можливо локально в хуці, якщо не потрібні іншим фічам; список усіх планів — у Redux, бо кешується), DTO для `GET /api/plans`, чек-лист станів для таблиці й селектора.
