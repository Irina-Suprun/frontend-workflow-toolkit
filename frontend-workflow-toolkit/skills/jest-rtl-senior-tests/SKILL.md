# Jest & React Testing Library — Senior Test Suite Generator

Скіл для генерації надійних, підтримуваних та поведінково-орієнтованих тестів для стеку
React / Next.js / Redux Toolkit / TypeScript. Мова відповіді користувачу — українська
(якщо користувач явно не попросив іншою).

## Перед тим як писати тест

1. Визнач **тип цілі**: презентаційний компонент / компонент із формою / компонент зі
   списком / компонент з async-станами (loading/success/error) / кастомний хук / Redux
   slice / Redux selector / thunk.
2. Відкрий відповідний референс-файл (див. "Де шукати шаблони" нижче) — не вигадуй
   структуру заново, використовуй перевірені паттерни та адаптуй їх під конкретний код.
3. Якщо користувач надав сам компонент/хук/слайс — прочитай його уважно: пропси, типи,
   селектори, side-effects, зовнішні залежності (next/navigation, fetch, Redux hooks).
   Якщо коду немає в контексті — попроси його показати, не вигадуй API.
4. Якщо в наданому коді є натяки на API-ключі, токени чи паролі — одразу попередь
   користувача і замаскуй їх у виводі.

## Базові принципи (незмінні для будь-якого тесту)

- **Тестуємо поведінку, не деталі реалізації.** Шукаємо елементи так, як їх шукав би
  користувач: `getByRole` / `findByRole` → `getByLabelText` → `getByPlaceholderText` →
  `getByText`. `data-testid` — лише крайній випадок, коли немає доступного семантичного
  способу знайти елемент (і це варто прокоментувати в коді чому).
- **Ніколи не query по class-name чи тегу** (`container.querySelector('.foo')`) — це
  прив'язка до реалізації, а не до контракту компонента.
- **AAA-структура**: Arrange → Act → Assert, з порожнім рядком між блоками для читабельності.
- **Взаємодія користувача — тільки через `@testing-library/user-event`**, ніколи через
  `fireEvent` (окрім вузьких випадків, які user-event не покриває, напр. `fireEvent.scroll`).
- **Асинхронність** — тільки `findBy*` або `waitFor`. Жодних `setTimeout`/штучних sleep.
- **Назви тестів описують поведінку та результат**, не механіку, і пишуться
  англійською (див. правило вище):
  - ДОБРЕ: `it('shows an error message when the profile-loading request fails')`
  - ПОГАНО: `it('works correctly')` / `it('тест 1')` / `it('показує помилку')`
- **Групування**: `describe('<Назва> Component', ...)`, `describe('use<Назва> Hook', ...)`,
  `describe('<назва>Slice', ...)`, `describe('select<Назва>', ...)`.
- **Мова всього тестового коду — завжди англійська**, незалежно від мови спілкування
  з користувачем у чаті. Це стосується `describe`, `it`/`test` описів, коментарів
  всередині тестового файлу, назв змінних і повідомлень у fixtures:
  - `describe('UserCard Component', ...)`
  - `it('shows an error message when the API request to load the profile fails', ...)`
  - ПОГАНО: `it('показує помилку, якщо запит на завантаження профілю впав', ...)`
  Пояснення поза кодом (перед/після блоку коду в чаті) залишається українською —
  англійською пишеться лише сам вміст `.test.ts`/`.test.tsx` файлу.

## Файлові конвенції (узгоджено з фронтенд-стандартами користувача)

- Тестовий файл співіснує поруч із цільовим файлом: `ComponentName.test.tsx`,
  `useHookName.test.ts`, `slice.test.ts`, `selectors.test.ts`.
- Якщо компонент організований як `Component.tsx` + `Component.types.ts` +
  `Component.module.scss` + `index.ts` — тест імпортує компонент **через `index.ts`**
  цієї ж папки (`import { UserCard } from './UserCard'`), а не напряму з `Component.tsx`,
  щоб перевіряти публічний контракт, а не внутрішню структуру файлів.
- Кінці рядків — LF (macOS), без BOM, без trailing whitespace.
- Жодного `any` у фікстурах, мок-даних чи типах пропсів — типізуй усе явно через
  інтерфейси/типи, що імпортуються з `*.types.ts` цільового модуля, або локально
  оголошені `Mock<...>`-типи для тестового файлу.

## Мок-стратегія (детально — `references/mocks-and-setup.md`)

- **Redux у connected-компонентах**: НЕ мокати селектори. Обгортати компонент у
  `renderWithProviders` з реальним стором RTK, попередньо наповненим типізованим
  `preloadedState: Partial<RootState>`. Slice-редюсери та селектори тестуються окремо,
  чистими юніт-тестами, без рендеру UI.
- **Next.js**: мокати `next/navigation` (`useRouter`, `usePathname`, `useSearchParams`)
  на рівні файлу через `jest.mock(...)`, типізовано.
- **Мережа**: моки на рівні модуля (`jest.fn()`) для простих кейсів, або MSW-хендлери
  для сценаріїв з кількома запитами / перевіркою заголовків-запиту.
- **Cleanup**: `afterEach(() => { jest.clearAllMocks(); })` в кожному файлі — жодного
  протікання стану між тестами.

## Redux Toolkit — окремо (детально — `references/redux-testing.md`)

- **Slice-редюсери**: тест дефолтного (initial) стану + тест кожного екшена окремо
  (включно з edge-кейсами: порожній payload, повторний виклик, скидання стану).
- **Selectors**: перевірка коректності обчислення + тест мемоізації для `createSelector`
  (повторний виклик з тим самим інпутом не викликає перерахунку — рахуємо recompute count).
- **Thunks**: мокати `dispatch` та зовнішню залежність (API-клієнт), перевіряти
  послідовність `pending → fulfilled/rejected` дій, без реального network-виклику.

## Кастомні хуки (детально — `references/hooks-testing.md`)

- Використовувати `renderHook` з `@testing-library/react`.
- Оновлення стану всередині хука — обов'язково через `act()`, якщо це не вже
  всередині `waitFor`/`findBy*` (які самі це обгортають).
- Хуки, що читають Redux, тестуються через `renderHook` з обгорткою-провайдером
  (та ж `renderWithProviders`, але для хуків, не для UI).

## Обов'язкове покриття (checklist перед тим, як показати тест користувачу)

Для **кожного** тестованого модуля перевір, що є тест-кейси на:

- [ ] Happy path — нормальний потік із валідними даними.
- [ ] Edge cases — порожній масив/список, відсутні опціональні пропси, дуже довгий
      текст, межові числові значення (0, негативні, дуже великі).
- [ ] Async-стани — Loading / Success / Error (якщо модуль їх має).
- [ ] Доступність неявно перевіряється самим використанням `getByRole`/`getByLabelText` —
      якщо тест не може знайти елемент через роль/лейбл, це сигнал про проблему a11y
      в самому компоненті, а не привід переходити на `testid`. Повідом про це користувачу.
- [ ] Для форм — валідація (некоректне значення → повідомлення про помилку) та
      успішний сабміт із правильними аргументами викликаного handler'а.

Не тестуємо надлишково: внутрішні деталі рендеру, кількість викликів `console.log`,
точні class-names, snapshot-тести за замовчуванням (тільки якщо користувач явно попросить).

## Формат виводу

Кожен тестовий файл, який ти показуєш користувачу, повинен містити:

1. Повні імпорти (React — якщо потрібен, RTL-утиліти, `userEvent`, типи, ціль тестування).
2. Мок-налаштування одразу після імпортів (jest.mock для next/navigation тощо).
3. Типізовані фікстури/мок-дані без `any`.
4. `describe`-блок(и) з `it`, що покривають чеклист вище.
5. `afterEach`/`beforeEach` з очищенням моків.

Це має бути **готовий до копіювання локальний код** — без пропозицій запускати
`npm test`, деплоїти чи пушити. Користувач працює виключно в ручному режимі: покажи
код, за потреби коротко поясни, що він перевіряє, і зупинись.

## Де шукати шаблони (references/)

- `references/component-testing.md` — презентаційний компонент, форма з валідацією,
  список (порожній/заповнений), компонент з async-станами (loading/success/error).
- `references/hooks-testing.md` — простий хук зі станом, async-хук (fetch/SWR-подібний),
  хук, залежний від Redux.
- `references/redux-testing.md` — slice reducer, selector (+ тест мемоізації), thunk.
- `references/mocks-and-setup.md` — повний код `renderWithProviders`, мок `next/navigation`,
  налаштування MSW-сервера для тестів.

Відкривай лише той файл, який відповідає поточному завданню — не тягни все одразу.
