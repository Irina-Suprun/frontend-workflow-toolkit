# Linear Task Manager

Перетворює обговорення (фіча, баг, архітектурний план) на структурований, готовий до створення Linear issue — і веде його по життєвому циклу: створення → гілка → старт роботи.

## Чому це важливо

Хороший issue економить час команди на ревʼю та онбордингу: description одразу дає контекст, технічні деталі й критерії готовності, без здогадок. Мета цього скіла — щоразу видавати issue такого рівня, а не просто "запиши в Linear те, що я сказав".

## Крок 0: Знайти команду (team)

Перед першим створенням issue в розмові з'ясуй, в яку Linear-команду писати:

1. Виклич `Linear:list_teams`.
2. Якщо команда одна — використовуй її мовчки, без питань до користувача.
3. Якщо команд декілька — запитай користувача, яку команду використати, і запам'ятай вибір на решту розмови (не питай повторно кожного разу).

## Крок 1: Зібрати контекст

Витягни з обговорення (повідомлення користувача, попередній план з architecture-planner, опис бага, стектрейс) інформацію, потрібну для шаблону нижче. Якщо чогось критичного не вистачає — короткого технічного контексту, конкретного компонента/файлу, чіткого acceptance-критерію — запитай користувача, а не вигадуй.

## Крок 2: Формат issue

Завжди формуй issue за цим шаблоном:

### Title
Короткий, у наказовому способі, з контекстом у квадратних дужках.
Приклад: `[Auth] Реалізувати доступну валідацію форми логіну`

### Description
```markdown
## Контекст та бізнес-ціль
[Що потрібно зробити і чому — 1-3 речення]

## Технічні деталі
- Компоненти/хуки/слайси: [...]
- Файлові шляхи (за colocation-структурою): [...]
- API-ендпоінти / DTO (якщо є): [...]

## Критерії готовності
- [ ] Функціональні вимоги виконані
- [ ] Без `any` / без `enum`
- [ ] Доступність (a11y) і клавіатурна навігація перевірені
- [ ] Unit-тести (Jest) написані

## Дизайн / специфікації
[Посилання або нотатки, якщо є]
```

Критерії готовності відображають прийняті в проєкті стандарти (сувора типізація, a11y, Jest) — не прибирай жоден пункт без явного прохання користувача, але можеш додати специфічні для конкретної задачі пункти.

## Крок 3: Підтвердження ПЕРЕД створенням

Ніколи не викликай `Linear:save_issue` одразу. Спершу покажи користувачу повністю відформатований Markdown issue (title + description) у чаті і дочекайся явного підтвердження ("створюй", "ок", "так"). Це не бюрократія заради бюрократії — Ірина працює виключно в ручному режимі і сама вирішує, що саме потрапляє у трекер.

Якщо користувач просить правки — внеси їх і покажи оновлену версію знову перед створенням.

## Крок 4: Створення issue

Після підтвердження:
1. Виклич `Linear:save_issue` з обраною командою, title та description.
2. Отримай ID створеного issue (наприклад `IR-102`).
3. Запропонуй git-гілку на основі ID: `<ISSUE-ID>-короткий-слаг-з-title`, наприклад `IR-102-login-form-validation`. Це лише пропозиція назви — сам скіл ніколи не виконує git-команди (`git checkout -b ...` користувач запускає власноруч).
4. Якщо в цій же розмові щойно було зроблено architecture-planner, git-workflow або styling-expert/senior-frontend-standards ревʼю — можеш згадати, що ID issue варто додати в майбутній commit-message (це вже сфера skill `git-workflow`).

## Крок 5: Старт роботи над issue

Коли користувач каже, що починає роботу над конкретним issue ("почала IR-102", "беруся за цю задачу"):

1. Виклич `Linear:list_issue_statuses` для команди цього issue, щоб отримати РЕАЛЬНІ назви статусів workflow (вони відрізняються між командами — не вважай, що "In Progress" існує саме так).
2. Знайди статус, що семантично відповідає "в роботі" (наприклад "In Progress", "In Dev", "Doing").
3. Запропонуй користувачу перевести issue саме в цей статус, назвавши його точну назву з Linear. Дочекайся підтвердження.
4. Після "так" — виклич `Linear:save_issue` (оновлення статусу) для переходу.

Не вгадуй назву статусу з памʼяті і не переводь issue без явного "так" від користувача — статуси відрізняються по командам, а помилковий перехід статусу вводить команду в оману.

## Приклад формату (референс)

**Вхід (обговорення):** "Потрібно зробити валідацію форми логіну доступною — зараз скрінрідер не читає помилки, і фокус не переходить на invalid-поле."

**Вихід (issue, показаний користувачу на підтвердження):**

```markdown
# [Auth] Реалізувати доступну валідацію форми логіну

## Контекст та бізнес-ціль
Форма логіну не повідомляє користувачам скрінрідерів про помилки валідації,
фокус не переміщується на невалідне поле. Потрібно привести форму
до стандартів a11y.

## Технічні деталі
- Компоненти: `LoginForm/LoginForm.tsx`
- Хуки: `useLoginFormValidation`
- Файлові шляхи: `src/features/auth/components/LoginForm/`

## Критерії готовності
- [ ] Помилки валідації озвучуються скрінрідером (aria-live / aria-describedby)
- [ ] Фокус переходить на перше невалідне поле після сабміту
- [ ] Без `any` / без `enum`
- [ ] Unit-тести (Jest) на хук валідації написані

## Дизайн / специфікації
—
```

## Межі скіла

- Скіл ніколи не виконує git-команди та не пропонує деплой/пуш — тільки назву гілки як текст.
- Скіл ніколи не створює і не оновлює issue без явного підтвердження користувача на конкретному кроці.
- Якщо в описі коду чи задачі трапляються натяки на API-ключі, токени чи паролі — не вставляй їх в issue, попередь користувача і замаскуй значення.
